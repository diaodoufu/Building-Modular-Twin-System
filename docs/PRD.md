# BMTS - 产品需求文档 (PRD)

> Building Modular Twin System - 建筑模块化孪生系统
> 版本：v1.0 | 日期：2026-07-11

---

## 1. 产品概述

### 1.1 产品名称
BMTS（Building Modular Twin System）- 建筑模块化孪生系统

### 1.2 产品定位
基于开放空间数据规范的跨尺度建筑孪生平台，让任何建筑空间都可以被描述、可视化、智能管理。

### 1.3 项目性质
学习/练手项目，按开源软件开发规范流程推进，目标：
1. 在过程中学习全栈开发和3D可视化技术
2. 做出一个完整可用的作品
3. 打造个人技术名片（开源展示 / 面试作品）

### 1.4 一句话描述
一个基于开放空间数据规范的跨尺度建筑孪生平台，以3D模型为入口，支持空间占用查询、智能预约、资产管理和数据分析。

### 1.5 终局愿景
最终形态是一个**多组织SaaS平台**：用户可以加入不同的顶级组织（学校、医院、甚至国家级机构），组织之间可自由切换。进入组织后，根据权限和层级粒度，从宏观全景细化到个人可感知、可操作的信息（如"我的教室在哪""我的预约有哪些"）。这是一套**组织无关、空间类型无关**的通用平台，校园只是第一个切入场景。

## 2. 核心设计理念

### 2.1 分级容器模型

一切皆容器。每个容器包含下一级子容器，同时拥有固有属性和可扩展属性。

```
校园(Campus)
  └── 建筑(Building)
        └── 楼层(Floor)
              └── 房间(Room)
                    └── 设备/工位(Resource)
```

**核心规则**：
- 每一级都是同构的容器结构：`{ id, name, type, parentId, children[], baseAttrs{}, extAttrs{} }`
- `baseAttrs`：该级容器的固有属性（如建筑有"建成年份"，楼层有"层高"）
- `extAttrs`：可扩展的自定义属性（如教室可以有"投影仪"属性，宿舍可以有"床位数"属性）
- 省→市→区→机构的分级管理是同一模型在更大尺度的体现
- 组织（Organization）是最顶层的容器，用户可以加入多个组织并自由切换

### 2.2 三层创新架构

| 层级 | 核心内容 | 价值 |
|------|----------|------|
| **规范层** | BMTS Schema — 开放建筑空间数据规范 | 从"一个应用"升级为"一个标准" |
| **平台层** | 跨尺度无缝缩放的3D可视化引擎 | 统一交互体验，同构容器无缝切换 |
| **智能层** | 基于数据的AI推荐与预测 | 从"查询工具"升级为"智能助手" |

## 3. 目标用户

### 3.1 MVP阶段目标用户

| 角色 | 描述 | 核心需求 |
|------|------|----------|
| **师生** | 校园内的教师和学生 | 查找教室、查看占用状态、预约空间 |
| **管理员** | 基建处/后勤/场地管理员 | 管理房间分配、审核预约、录入数据、查看统计 |

### 3.2 未来扩展用户
- 访客：导航查找
- 领导层：查看校园整体运行数据和决策支持
- 政府/规划部门：宏观视角的区域规划（省→市→区架构）
- 多组织用户：同时属于多个组织（如既是某校教师又是某医院顾问），在组织间切换

## 4. 功能需求

### 4.1 Phase 1 — 平台层核心

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F1.1 | 分级容器数据模型 | 实现校园→建筑→楼层→房间的容器层级，支持固有属性+可扩展属性 | P0 |
| F1.2 | 3D场景渲染 | 渲染单栋建筑的3D精模，可旋转/缩放/平移 | P0 |
| F1.3 | 3D交互选择 | 点击楼层/房间高亮选中，弹出信息面板 | P0 |
| F1.4 | 房间信息面板 | 展示选中房间的完整属性信息（类型、面积、用途、设备等） | P0 |
| F1.5 | 建筑楼层切换 | 在3D视图中切换查看不同楼层 | P1 |
| F1.6 | 左侧导航树 | 树形结构展示校园→建筑→楼层→房间的层级 | P1 |

### 4.2 Phase 2 — 功能层

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F2.1 | 房间占用状态 | 在3D模型和列表中展示房间状态（空闲/使用中/已预约） | P0 |
| F2.2 | 空间预约 | 师生选择空闲房间发起预约请求 | P0 |
| F2.3 | 预约审核 | 管理员审核预约请求（通过/拒绝） | P0 |
| F2.4 | 我的预约 | 查看个人预约记录和状态 | P1 |
| F2.5 | 数据录入 | 管理员手动录入/编辑建筑、楼层、房间信息 | P0 |
| F2.6 | 用户认证 | 登录/注册，区分师生和管理员角色 | P0 |

### 4.3 Phase 3 — 规范层

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F3.1 | BMTS Schema定义 | 定义开放建筑空间数据规范的JSON Schema | P0 |
| F3.2 | 数据导入 | 按Schema规范导入JSON/YAML格式的建筑数据 | P0 |
| F3.3 | 数据导出 | 将当前数据按Schema规范导出 | P0 |
| F3.4 | Schema文档 | 提供可读的规范文档和使用指南 | P1 |
| F3.5 | 数据验证 | 导入时校验数据是否符合Schema规范 | P1 |

### 4.4 Phase 4 — 智能层

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F4.1 | 空间利用率统计 | 按建筑/楼层/类型统计空间使用率 | P0 |
| F4.2 | 预约趋势分析 | 预约量随时间的变化趋势 | P1 |
| F4.3 | 智能预约推荐 | 基于历史数据推荐最优预约时间/地点 | P1 |
| F4.4 | 异常占用检测 | 检测长期占用但未使用、预约未到场等异常 | P2 |

### 4.5 Phase 5 — 跨尺度缩放

| 编号 | 功能 | 描述 | 优先级 |

### 4.6 Phase 6 — 多组织与细粒度权限

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|

| F6.7 | 移动端适配 | 响应式布局，手机浏览器可用，3D场景简化为2D平面图 | P1 |

## 5. 非功能需求

| 编号 | 需求 | 描述 |
|------|------|------|
| NF1 | 性能 | 单栋建筑（约50个房间）3D渲染帧率 ≥ 30fps |
| NF2 | 响应时间 | 页面加载 ≤ 3s，API响应 ≤ 500ms |
| NF3 | 兼容性 | 支持Chrome、Edge、Firefox最新版（PC+移动端浏览器） |
| NF4 | 数据规模 | 支持单个校园 ≤ 20栋建筑，每栋 ≤ 10层，每层 ≤ 30个房间 |
| NF5 | 可扩展性 | 数据模型支持自定义扩展属性，无需改代码 |
| NF6 | 多组织 | 单个用户可加入 ≤ 10个组织，组织间数据隔离 |
| NF7 | 移动端 | 响应式布局，核心功能（查询、预约、我的）在手机浏览器可用 |

## 6. 用户故事

### 6.1 师生用户故事

- 作为一个学生，我想在3D地图上找到某间教室的位置，这样我能快速找到上课地点
- 作为一个学生，我想查看某个教室当前是否空闲，这样我可以去自习
- 作为一个学生，我想预约一个空闲的讨论室，这样我可以在特定时间使用
- 作为一个教师，我想查看我预约的所有房间，这样我可以管理我的教学安排

### 6.2 管理员用户故事

- 作为一个管理员，我想在3D地图上直观看到所有房间的占用状态，这样我可以了解空间利用情况
- 作为一个管理员，我想审核预约请求，这样我可以管控空间使用
- 作为一个管理员，我想录入和编辑房间信息，这样我可以保持数据准确
- 作为一个管理员，我想查看空间利用率统计，这样我可以优化空间分配

### 6.3 多组织用户故事（终局）

- 作为一个用户，我想搜索并加入我的学校/医院的组织，这样我可以查看和使用该组织的空间信息
- 作为一个用户，我想在我加入的多个组织之间快速切换，这样我不需要重复登录
- 作为一个用户，我想看到与我相关的个人化信息（我的教室、我的预约），这样我不用在大量数据中寻找
- 作为一个管理员，我想限制某个用户只能看到特定楼层/房间，这样我可以控制信息访问范围
