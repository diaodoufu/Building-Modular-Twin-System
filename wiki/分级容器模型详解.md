# 分级容器模型详解

## 核心概念

一切皆容器。BMTS 的核心数据模型是分级容器（Hierarchical Container），每个容器包含下一级子容器，同时拥有固有属性和可扩展属性。

## 容器层级

```
校园 (Campus)
  └── 建筑 (Building)
        └── 楼层 (Floor)
              └── 房间 (Room)
                    └── 设备/工位 (Resource)
```

## 同构结构

所有层级的容器共享统一的数据结构：

```json
{
  "id": "uuid",
  "name": "容器名称",
  "type": "campus|building|floor|room|resource",
  "parentId": "父容器UUID",
  "children": [],
  "baseAttrs": {},
  "extAttrs": {},
  "position": { "x": 0, "y": 0, "z": 0 },
  "dimensions": { "width": 0, "height": 0, "depth": 0 }
}
```

### baseAttrs（固有属性）

每种类型的容器有固定的固有属性：

| type | baseAttrs 字段 |
|------|----------------|
| campus | `{ "address": "", "area_sqm": 0, "description": "" }` |
| building | `{ "built_year": 0, "total_floors": 0, "area_sqm": 0, "address": "" }` |
| floor | `{ "level": 0, "height_m": 0, "area_sqm": 0 }` |
| room | `{ "room_number": "", "area_sqm": 0, "capacity": 0, "room_type": "classroom|office|lab|dormitory|meeting|utility" }` |
| resource | `{ "resource_type": "desk|projector|whiteboard|bed", "status": "available|in_use|maintenance" }` |

### extAttrs（可扩展属性）

用户可自由添加的自定义属性，无需改代码。示例：

```json
// 教室的扩展属性
{
  "hasProjector": true,
  "hasWhiteboard": true,
  "airConditioning": true
}

// 宿舍的扩展属性
{
  "bedCount": 4,
  "hasBathroom": true,
  "floorType": "tile"
}
```

## 数据库设计

容器表使用 PostgreSQL 的 JSONB 字段存储 baseAttrs 和 extAttrs，支持 GIN 索引实现高效查询：

```sql
CREATE TABLE containers (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    org_id      UUID NOT NULL,
    type        VARCHAR(32) NOT NULL,
    name        VARCHAR(128) NOT NULL,
    parent_id   UUID REFERENCES containers(id),
    sort_order  INTEGER DEFAULT 0,
    base_attrs  JSONB NOT NULL DEFAULT '{}',
    ext_attrs   JSONB NOT NULL DEFAULT '{}',
    position    JSONB,
    dimensions  JSONB,
    created_at  TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at  TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

## 3D 可视化

每种容器在 3D 场景中有对应的渲染方式：

- **Campus**：俯视图，建筑分布在地面上
- **Building**：3D 长方体，颜色按类型区分
- **Floor**：水平切片，可上下切换
- **Room**：颜色编码表示状态（空闲/占用/已预约）
- **Resource**：房间内的标记点

## 为什么选择这个模型

1. **通用性**：同构结构意味着同一套 CRUD 代码适用于所有层级
2. **可扩展**：extAttrs 让用户无需改代码就能添加新属性
3. **跨尺度**：从省→市→区→机构的分级管理是同一模型在更大尺度的体现
4. **类型安全**：baseAttrs 保障核心数据结构一致，extAttrs 提供灵活性
